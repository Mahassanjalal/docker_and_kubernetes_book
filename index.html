01 Introduction to Containers Docker
E What Are Containers?

Containers are lightweight, standalone executable packages that include everything needed to run a piece of software — code, runtime, system tools, libraries, and settings. Unlike virtual machines, containers share the host operating system kernel and do not require a full guest OS, making them extremely fast to start and efficient in resource usage.

Containers isolate processes from each other within a shared OS kernel. Each container runs in its own isolated user space with its own filesystem, network stack, and process tree. This isolation is achieved through Linux kernel features: namespaces (for isolation) and cgroups (for resource limiting).

A Real-World Analogy
Think of containers like shipping containers in logistics. Before shipping containers existed, goods were loaded individually — fragile items mixed with heavy machinery, different shapes and sizes causing chaos. Shipping containers standardized everything: same dimensions, same interface, works on any ship, truck, or train. Similarly, software containers package your application with all its dependencies into a standard unit that runs consistently on any infrastructure — your laptop, a test server, or cloud production.

02 Docker Architecture Docker
E Core Components

Docker uses a client-server architecture with three main components:

  • Docker Client (docker CLI) — The command-line interface you interact with. It sends commands to the Docker daemon via REST API.
  • Docker Daemon (dockerd) — The background service that listens for Docker API requests, manages images, containers, networks, and volumes.
  • Docker Registry — A storage and distribution system for Docker images. Docker Hub is the default public registry.

05 Docker Volumes Docker
E Persistent Storage Explained

Containers are ephemeral — when a container is removed, all data written to its filesystem is lost. Volumes provide a mechanism to persist data beyond the container's lifecycle. Docker offers three storage options:

  • Volumes — Managed by Docker, stored in /var/lib/docker/volumes/. Best for production.
  • Bind mounts — Map a specific host path into the container. Best for development.
  • tmpfs mounts — Stored in host memory only. Best for sensitive temporary data.
D Storage Options Comparison
Docker Storage Types
> Volume Commands
$ docker volume create my-data # Create a named volume
$ docker volume ls # List all volumes
$ docker volume inspect my-data # Show volume details (mountpoint, driver)
$ docker run -d -v my-data:/var/lib/mysql mysql:8 # Mount volume into container
$ docker run -d -v $(pwd)/data:/app/data nginx # Bind mount: host dir → container
$ docker run -d --tmpfs /tmp:rw,size=100m nginx # tmpfs mount (RAM only)
$ docker volume rm my-data # Remove volume (must not be in use)
$ docker volume prune # Remove all unused volumes

A Real-World Analogy
Imagine you're working in a hotel room (container). Everything in the room is temporary — when you check out, it's all cleaned away. A volume is like a safety deposit box at the front desk — managed by the hotel, persists after you leave, and you can access it from any room. A bind mount is like bringing your own filing cabinet from home and placing it in the room — you control exactly where it is and what's in it. A tmpfs is like a whiteboard in the room — great for quick notes, but erased when you leave.

Persisting Database Data

  1. Run a PostgreSQL container WITHOUT a volume: docker run -d --name db-no-vol -e POSTGRES_PASSWORD=test postgres:16-alpine
  2. Create a database and table inside it
  3. Stop and remove the container — data is gone

! Troubleshooting Tips
"Volume is already in use" on rm — Stop and remove the container using it first, or use docker rm -f container_name
Permission denied writing to bind mount — The container process UID/GID must match the host directory permissions. Use :z or :Z suffix for SELinux relabeling.
Bind mount path not found — Docker auto-creates the directory if it doesn't exist, but as root. Create it beforehand with correct ownership.
* Best Practices
  • Use named volumes for production data — never bind mounts for databases
  • Use bind mounts with :ro for configuration file injection
  • Use volume drivers (local, NFS, EBS) for cloud-native storage
  • Never mount sensitive files (SSH keys, certs) as bind mounts without :ro
  • Regularly back up volumes: docker run --rm -v mydata:/data -v $(pwd):/backup alpine tar czf /backup/backup.tar.gz /data
06 Docker Compose Docker
E What Is Docker Compose?

Docker Compose is a tool for defining and running multi-container Docker applications. With a single docker-compose.yml file, you configure all your services, networks, and volumes, then start everything with one command: docker compose up.

Compose handles service dependencies, networking, and orchestrated startup/shutdown order. It's the standard tool for local development environments and can also be used for small-scale production deployments.

07 Multi-stage Builds Docker
E Why Multi-stage Builds?

Multi-stage builds allow you to use multiple FROM statements in a single Dockerfile. Each FROM begins a new build stage. You can selectively copy artifacts from one stage to another, discarding everything else. This dramatically reduces final image size by excluding build tools, source code, and intermediate files.

D Multi-stage Build Flow
Stage Flow Diagram
C Go Application Multi-stage Build
DockerfileGo Multi-stage
# ==================== BUILD STAGE ====================
FROM golang:1.22-alpine AS builder

WORKDIR /src

# Cache dependencies separately
COPY go.mod go.sum ./
RUN go mod download

# Copy source and build
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build \
    -ldflags="-s -w" \
    -o /app/server ./cmd/server

# ==================== RUNTIME STAGE ====================
FROM alpine:3.19

# Install ca-certificates for HTTPS (if needed)
RUN apk add --no-cache ca-certificates tzdata

COPY --from=builder /app/server /usr/local/bin/server

EXPOSE 8080

ENTRYPOINT ["server"]
C Node.js Multi-stage Build
DockerfileNode.js Multi-stage
# Stage 1: Build frontend assets
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build  # Outputs to /app/dist

# Stage 2: Production runtime (slim image)
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production && npm cache clean --force
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/server.js ./
USER node
EXPOSE 3000
CMD ["node", "server.js"]
A Real-World Analogy
A multi-stage build is like a restaurant kitchen. The prep kitchen (build stage) has all the heavy equipment — industrial mixers, food processors, cutting boards, raw ingredients. The final dish (runtime stage) is just the plated meal on a clean plate. You don't bring the entire kitchen to the customer's table — only the finished product. The build stage is discarded, keeping the final image lean.
> Build Commands
$ docker build --target builder -t myapp:build . # Build only the builder stage (for debugging)
$ docker build -t myapp:1.0 . # Build all stages, final image = last stage
$ docker build --no-cache -t myapp:1.0 . # Force rebuild all stages

  • Copy only the exact artifacts needed — never copy the entire build directory
  • Use -ldflags="-s -w" for Go/Rust to strip debug symbols
  • Keep the final runtime image as minimal as possible (scratch, distroless, alpine)

Container security requires a defense-in-depth approach. The main threat vectors include: privileged container escape, vulnerable base images, secrets in images, supply chain attacks, and misconfigured permissions. Each layer of the container stack needs protection.

! Troubleshooting Tips
"Permission denied" after switching to non-root — The files copied during build (as root) aren't writable by the new user. Add RUN chown -R uid:gid /app before USER.

* Best Practices
  • Never run as root — always use USER directive or distroless/nonroot images
  • Never use --privileged flag — it removes ALL kernel protections
  • Use --cap-drop ALL and --cap-add only required capabilities
  • Use --read-only filesystem with --tmpfs for writable directories
  • Scan images in CI/CD pipelines before pushing to registries
  • Never embed secrets in images — use runtime environment variables or secret managers
  • Enable Docker Content Trust for image signature verification
  • Use rootless Docker mode to eliminate daemon-as-root risk
09 Kubernetes Architecture Kubernetes
E What Is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration platform that automates deployment, scaling, and management of containerized applications. It provides a declarative configuration model where you describe the desired state, and Kubernetes works to achieve and maintain that state.

At its core, Kubernetes solves the problems that arise when you need to run hundreds or thousands of containers across multiple machines: scheduling, self-healing, service discovery, load balancing, rolling updates, and secret/configuration management.

A Pod is the smallest deployable unit in Kubernetes. It represents a single instance of a running process in your cluster. 11 ReplicaSets Kubernetes
E What Is a ReplicaSet?

A ReplicaSet ensures that a specified number of pod replicas are running at any given time. It constantly monitors the cluster and creates or deletes pods to match the desired replica count. If a node fails and pods are lost, the ReplicaSet creates replacements on healthy nodes.

Important: You rarely create ReplicaSets directly. Instead, you use Deployments, which manage ReplicaSets for you and add rolling update capabilities.

D ReplicaSet Reconciliation Loop
ReplicaSet Reconciliation Flow

kind: ReplicaSet
metadata:
  name: nginx-rs
spec:

  replicas: 3

  # Selector determines which pods this RS manages
  selector:
    matchLabels:
      app: nginx

    metadata:

        app: nginx  # MUST match selector
    spec:
      containers:
      - name: nginx
        image: nginx:1.25-alpine
        ports:
        - containerPort: 80

A Real-World Analogy
A ReplicaSet is like a shift supervisor at a factory who is told "I need exactly 3 workers on the assembly line at all times." If a worker takes a break or leaves, the supervisor immediately hires a replacement. The supervisor doesn't care WHICH 3 workers — only that exactly 3 are present. The selector (labels) is how the supervisor identifies who belongs to their team (blue uniforms = my team).

12 Deployments Kubernetes
E What Is a Deployment?

A Deployment is a higher-level abstraction over ReplicaSets that adds declarative updates, rolling changes, and rollback capabilities. When you create a Deployment, it creates a ReplicaSet, which in turn creates Pods. When you update the Deployment (e.g., change the image version), it creates a new ReplicaSet and gradually transitions traffic.

Deployments are the most common way to run stateless applications in Kubernetes.

D Deployment → ReplicaSet → Pod Hierarchy
Object Hierarchy
C Complete Deployment Manifest
deployment.yamlKubernetes YAML
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  labels:
    app: web-app
spec:
  replicas: 3

  # Rolling update strategy
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # Can create 1 extra pod above desired
      maxUnavailable: 0  # No pods below desired count

  selector:
    matchLabels:
      app: web-app

  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
      - name: web-app
        image: myapp:1.2.0
        ports:
        - containerPort: 3000
        resources:
          requests:
            cpu: "100m"
            memory: "128Mi"
          limits:
            cpu: "500m"
            memory: "256Mi"
        livenessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 10
          periodSeconds: 15
        readinessProbe:
          httpGet:
            path: /ready
            port: 3000
          initialDelaySeconds: 5
          periodSeconds: 5

E What Are ConfigMaps?

ConfigMaps store non-confidential configuration data as key-value pairs. They decouple configuration from container images, allowing the same image to be used across different environments (dev, staging, prod) by simply swapping the ConfigMap.

ConfigMaps can be consumed as: environment variables, command-line arguments, or mounted files in volumes.

18 Ingress Kubernetes
E What Is Ingress?

An Ingress is an API object that manages external access to services in a cluster, typically HTTP/HTTPS. It provides: name-based virtual hosting (app1.example.com → svc1, app2.example.com → svc2), path-based routing (/api → backend, / → frontend), TLS termination, and load balancing — all through a single external IP.

D Ingress Architecture
Ingress Traffic Flow
C Ingress Manifest with TLS
ingress.yamlKubernetes YAML
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /$2
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/rate-limit: "10"
spec:
  ingressClassName: nginx

  # TLS configuration
  tls:
  - hosts:
    - app.example.com
    - api.example.com
    secretName: tls-certificate

  rules:
  - host: app.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-svc
            port:
              number: 80
  - host: api.example.com
    http:
      paths:
      - path: /api(/|$)(.*)
        pathType: ImplementationSpecific
        backend:
          service:
            name: backend-svc
            port:
              number: 3000

$ kubectl get ingress # List ingresses

* Best Practices
  • Always enable TLS — never serve HTTP to production traffic
  • Use cert-manager for automatic TLS certificate provisioning (Let's Encrypt)
  • Set explicit ingressClassName instead of relying on default annotations
  • Use pathType: Prefix or Exact — avoid ImplementationSpecific for predictable behavior
  • Add rate limiting and connection limits at the Ingress level
! Troubleshooting Tips

19 Resource Requests & Limits Kubernetes
E Why Resource Management Matters

Kubernetes uses requests (guaranteed minimum resources for scheduling) and limits (maximum resources a container can consume) to make scheduling decisions and prevent noisy neighbor problems. Without requests, the scheduler can't make intelligent placement decisions. Without limits, a single container can consume all node resources and crash other pods.

D Requests vs Limits
How the Scheduler Uses Requests and Limits
C Resource Configuration
resources.yamlKubernetes YAML
spec:
  containers:
  - name: app
    image: myapp:1.0
    resources:
      # Guaranteed minimum — used for scheduling decisions
      requests:
        cpu: "250m"      # 250 millicores = 0.25 CPU cores
        memory: "256Mi"   # 256 Mebibytes
      # Hard ceiling — container is throttled (CPU) or OOMKilled (memory)
      limits:
        cpu: "500m"      # 0.5 CPU cores max
        memory: "512Mi"   # 512 MiB max — exceeded = OOMKill
> Resource Commands
$ kubectl top nodes # Node resource usage (requires metrics-server)
$ kubectl top pods # Pod resource usage
$ kubectl describe node node-1 | grep -A5 "Allocated" # Resource allocation summary
* Best Practices
  • Always set both requests AND limits on every container
  • Set requests = limits for QoS class Guaranteed (most predictable)
  • Use LimitRange defaults at namespace level so pods without resources get sensible defaults
  • Use ResourceQuota to limit total namespace consumption
  • Profile your app's actual usage before setting values — don't guess
  • Memory: if a container exceeds its limit, it gets OOMKilled. CPU: it gets throttled, not killed.
! Troubleshooting Tips
OOMKilled (Exit Code 137) — Container exceeded memory limit. Increase the memory limit or fix the memory leak.
Pod stuck in "Pending" — "Insufficient cpu" — No node has enough unrequested CPU/memory. Add nodes, remove unused pods, or reduce requests.
CPU throttling — slow performance — CPU usage hitting the limit. Check with kubectl top pods and compare against limits.
20 Liveness & Readiness Probes Kubernetes
E Probe Types

Probes are health checks that Kubernetes uses to understand your container's state. There are three types:

  • Liveness Probe — "Is the process alive?" If it fails, K8s restarts the container. Catches deadlocks and infinite loops.
  • Readiness Probe — "Is it ready to serve traffic?" If it fails, the pod is removed from Service endpoints. Catches startup delays and temporary unavailability.
  • Startup Probe — "Has it finished starting?" Disables liveness/readiness checks until it succeeds. For slow-starting containers (large apps, JVMs).
D Probe Decision Flow
Probe Decision Tree
C Probe Configuration
probes.yamlAll Three Probe Types
spec:
  containers:
  - name: app
    image: myapp:1.0
    ports:
    - containerPort: 8080

    # Startup probe: give app 60 seconds to start
    startupProbe:
      httpGet:
        path: /started
        port: 8080
      failureThreshold: 12   # 12 failures × 5s period = 60s max startup
      periodSeconds: 5

    # Liveness: restart if unhealthy for 30 seconds
    livenessProbe:
      httpGet:
        path: /health
        port: 8080
      initialDelaySeconds: 15
      periodSeconds: 10
      failureThreshold: 3   # 3 failures → restart

    # Readiness: remove from service if not ready
    readinessProbe:
      httpGet:
        path: /ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
      failureThreshold: 3

    # Alternative: TCP probe (just check if port is open)
    # livenessProbe:
    #   tcpSocket:
    #     port: 8080

    # Alternative: Exec probe (run a command)
    # livenessProbe:
    #   exec:
    #     command: ["pg_isready", "-U", "admin"]
* Best Practices
  • Always implement separate /health and /ready endpoints — they serve different purposes
  • Use startup probes for slow-starting apps (Java, ML models) to prevent premature restarts
  • Set initialDelaySeconds based on actual startup time — too short causes CrashLoopBackOff
  • Liveness probes should NOT depend on external dependencies — only check the process itself
  • Readiness probes CAN check external dependencies (DB connection, cache) to temporarily remove from traffic
! Troubleshooting Tips
CrashLoopBackOff with liveness probe failures — The app starts but the liveness probe kills it before it's ready. Increase initialDelaySeconds or add a startup probe.
Pod shows "Ready: 0/1" but is Running — Readiness probe is failing. Check if the endpoint exists and returns 200.

Deployments support two update strategies: RollingUpdate (default — gradually replaces old pods with new ones) and Recreate (kills all old pods before creating new ones — causes downtime but simpler). The rolling update is controlled by maxSurge (how many extra pods above desired count) and maxUnavailable (how many pods below desired count).

22 Horizontal Pod Autoscaler Kubernetes
E What Is the HPA?

The Horizontal Pod Autoscaler automatically scales the number of pod replicas based on observed CPU utilization, memory usage, or custom metrics. It adjusts the replica count of a Deployment, StatefulSet, or ReplicaSet to match demand, ensuring you have enough pods to handle traffic without over-provisioning.

HPA operates as a control loop: every 15 seconds (by default), it queries metrics from the metrics-server (for CPU/memory) or Prometheus Adapter (for custom metrics), compares against the target threshold, and adjusts replicas accordingly.

D HPA Decision Loop
HPA Scaling Decision Flow
minReplicas: 2 — never scale below 2 (even if CPU is 0%) • maxReplicas: 10 — never scale above 10 (cost protection) • stabilizationWindowSeconds: 300 — wait 5 min of low usage before scaling down (prevents flapping) • Cool-down: HPA won't scale up/down more than once per evaluation cycle • Scale-up is immediate; scale-down has stabilization delay to prevent thrashing
E HPA Calculation Example
MetricValue
Target CPU utilization50%
Current CPU utilization80%
Current replicas4
Desired replicasceil(80/50 × 4) = ceil(6.4) = 7

If maxReplicas is 6, it would be capped at 6. If minReplicas is 8, it stays at 8.

C HPA Manifest
hpa.yamlKubernetes YAML
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70  # Target 70% CPU
  - type: Resource
    resource:
      name: memory
      target:
        type: AverageValue
        averageValue: "500Mi"  # Target 500Mi avg memory
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # Wait 5min before scaling down
      policies:
      - type: Percent
        value: 10        # Remove max 10% of pods per cycle
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0    # Scale up immediately
      policies:
      - type: Percent
        value: 50        # Add max 50% of pods per cycle
        periodSeconds: 60
      - type: Pods
        value: 4          # Or add max 4 pods per cycle (whichever is lower)
        periodSeconds: 60
> HPA Commands
$ kubectl autoscale deployment web-app --cpu-percent=70 --min=2 --max=10 # Quick create HPA
$ kubectl get hpa # List all HPAs with current metrics
$ kubectl describe hpa web-app-hpa # Detailed HPA info including events

$ kubectl delete hpa web-app-hpa # Remove HPA (replicas stay at current count)
A Real-World Analogy
HPA is like a smart thermostat for your office building. If the temperature (CPU) rises above 70°, it turns on more AC units (pods). If it drops below, it gradually turns some off — but with a delay (stabilization window) to avoid constantly switching on and off. The min/max is like saying "always keep at least 2 AC units running, and never turn on more than 10."

.github/workflows/deploy.yamlGitHub Actions
name: Build and Deploy to Kubernetes

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ===== JOB 1: Build, Test, Scan, Push =====
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
    - name: Checkout code
      uses: actions/checkout@v4

    - name: Set up Docker Buildx
      uses: docker/setup-buildx-action@v3

    - name: Log in to Container Registry
      uses: docker/login-action@v3
      with:
        registry: ${{ env.REGISTRY }}
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}

    - name: Extract metadata (tags, labels)
      uses: docker/metadata-action@v5
      id: meta
      with:
        images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
        tags: |
          type=sha,prefix=
          type=ref,event=branch

    - name: Build and push Docker image
      uses: docker/build-push-action@v5
      with:
        context: .
        push: ${{ github.event_name != 'pull_request' }}
        tags: ${{ steps.meta.outputs.tags }}
        labels: ${{ steps.meta.outputs.labels }}
        cache-from: type=gha
        cache-to: type=gha,mode=max

    - name: Scan image with Trivy
      uses: aquasecurity/trivy-action@master
      with:
        image-ref: ${{ steps.meta.outputs.tags }}
        severity: CRITICAL,HIGH
        exit-code: '1'

  # ===== JOB 2: Deploy to K8s =====
  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    if: github.event_name == 'push'

    steps:
    - name: Checkout code
      uses: actions/checkout@v4

    - name: Configure kubectl
      uses: azure/setup-kubectl@v3

    - name: Set up kubeconfig
      run: |
        mkdir -p $HOME/.kube
        echo "${{ secrets.KUBE_CONFIG }}" | base64 -d > $HOME/.kube/config

    - name: Update image tag in deployment
      run: |
        sed -i "s|image:.*|image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:sha-$(echo ${{ github.sha }} | cut -c1-7)|" k8s/deployment.yaml

    - name: Deploy to Kubernetes
      run: |
        kubectl apply -f k8s/
        kubectl rollout status deployment/web-app -n production --timeout=300s

    - name: Verify deployment
      run: |
        kubectl get pods -n production -l app=web-app
        kubectl rollout history deployment/web-app -n production | tail -3
* Best Practices
  • Use Docker Buildx cache with GitHub Actions cache (cache-from: type=gha) to speed up builds
  • Tag images with git SHA, not "latest" — enables precise rollbacks
  • Run security scanning (Trivy, Snyk) in CI — block deployments on critical CVEs
  • Store kubeconfig as a base64-encoded secret — never commit it to the repo
  • Use separate environments: deploy to staging first, then promote to production
  • Consider GitOps (ArgoCD/Flux) instead of kubectl apply in CI — the CI just updates the repo, ArgoCD syncs
  • Use GitHub Environments for approval gates before production deployment

29 Troubleshooting Guide Production
E Systematic Troubleshooting Approach

When something goes wrong in Kubernetes, follow this systematic approach: Describe the symptom → Check the object (kubectl describe) → Read the logs (kubectl logs) → Check events → Verify configuration → Check networking → Check resources.

D Troubleshooting Flowchart
K8s Troubleshooting Decision Tree
! Common Issues Quick Reference
ImagePullBackOff — Wrong image name/tag, private registry without imagePullSecrets, registry unreachable. Fix: kubectl create secret docker-registry, verify image exists.
CrashLoopBackOff — App crashes on start. Fix: kubectl logs <pod> --previous. Common: missing env vars, port conflicts, syntax errors in config.
OOMKilled (Exit 137) — Memory limit exceeded. Fix: increase memory limit, fix memory leak. Check: kubectl describe pod → Last State → OOMKilled.
Pod stuck in ContainerCreating — Volume mount failing or image pull slow. Fix: kubectl describe pod → Events section.
Pod Pending — "Insufficient cpu" — No node has enough unrequested resources. Fix: add nodes, remove unused pods, reduce requests.
0/1 Ready, Running — Readiness probe failing. Fix: check probe endpoint, increase initialDelaySeconds, check dependent services.
Service has no endpoints — Selector labels don't match pod labels. Fix: kubectl get pods --show-labels vs service selector.
Ingress 503 — Service endpoint issue or wrong service name in Ingress. Fix: kubectl get endpoints, verify service name.
Ingress 404 — Path matching issue. Fix: check path and pathType in Ingress rules.
Node NotReady — kubelet issues. Fix: journalctl -u kubelet on the node, check network, disk, container runtime.
HPA not scaling — Missing CPU requests on containers or metrics-server not running. Fix: add requests, install metrics-server.
PVC Pending — No StorageClass matches or insufficient capacity. Fix: check kubectl describe pvc, verify StorageClass.

E Top Interview Questions
Docker Questions
QuestionKey Points
What's the difference between a Docker image and a container?Image = read-only template with layers. Container = running instance with writable layer on top.
How does Docker networking work?Bridge (default, isolated), Host (shares host net), Overlay (multi-host), None (no net). Custom bridge provides DNS.
What is a Docker volume vs bind mount?Volume: managed by Docker, stored in /var/lib/docker/volumes. Bind mount: maps host path, user-managed.
Explain multi-stage builds.Multiple FROM statements. Build stage compiles, runtime stage copies only binary. Reduces image from GBs to MBs.
How do you secure a Docker container?Non-root user, read-only FS, drop capabilities, no --privileged, scan images, no secrets in images, rootless Docker.
What is the difference between CMD and ENTRYPOINT?ENTRYPOINT defines the executable (hard to override). CMD provides default arguments (easy to override). Use both together.
How does copy-on-write work in Docker?Each layer is read-only. When a container modifies a file, it copies it to the writable layer. Unmodified files are read from lower layers.
Kubernetes Questions
QuestionKey Points
What's the difference between a Pod and a Deployment?Pod: smallest unit, 1+ containers, ephemeral. Deployment: manages ReplicaSets, provides rolling updates, rollbacks, declarative scaling.
Explain the Kubernetes architecture.Control plane (API server, etcd, scheduler, controllers) + Worker nodes (kubelet, kube-proxy, container runtime). All communication through API server.
What happens when you create a Pod?kubectl → API server → etcd (stored) → scheduler (binds to node) → kubelet (creates container via CRI) → kube-proxy (updates network rules).
What are the Service types?ClusterIP (internal), NodePort (node IP:port), LoadBalancer (cloud LB), ExternalName (DNS CNAME), Headless (clusterIP: None).
How does a rolling update work?Deployment creates new RS, gradually scales up new pods and down old pods based on maxSurge/maxUnavailable. Readiness probe gates traffic.
What's the difference between a ConfigMap and a Secret?ConfigMap: non-sensitive config, stored in plain text in etcd. Secret: sensitive data, base64-encoded (not encrypted unless you configure it).
Explain HPA.Auto-scales replicas based on metrics (CPU, memory, custom). Requires resource requests. Uses formula: desired = ceil(current/target × currentReplicas).
When would you use a StatefulSet vs Deployment?StatefulSet: stable identity (ordered naming), stable storage (per-pod PVC), ordered scaling. Use for DBs, queues. Deployment: stateless apps.
What is an Ingress?L7 HTTP/HTTPS routing. Single IP for multiple services. Name-based and path-based routing. TLS termination. Requires Ingress Controller.
What are liveness vs readiness probes?Liveness: restart container if fails (catches deadlocks). Readiness: remove from Service endpoints if fails (catches not-ready state). Both critical.
Explain RBAC in Kubernetes.Role/ClusterRole define permissions. RoleBinding/ClusterRoleBinding bind roles to subjects (users, groups, service accounts). Namespace-scoped vs cluster-scoped.
What is a PodDisruptionBudget?Limits voluntary disruptions (node drain,升级). Ensures minAvailable or maxUnavailable pods during maintenance operations.
